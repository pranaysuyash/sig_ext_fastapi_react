# 30/60/90 Roadmap: Signature Field Identification and Native PDF Editing

**Project:** Signature Extractor / SignKit  
**Date:** 2026-06-17  
**Status:** Draft execution roadmap  
**Scope:** The first three months of expansion work after the research pass.

## Goal

Turn the current signature extractor into a PDF-aware document workflow app without losing the existing extraction flow.

The order matters:

1. Detect where signatures belong.
2. Place extracted signatures into PDFs.
3. Save edited PDFs safely.
4. Expand into workflow automation and layout intelligence.

## Month 1: Foundation And Discovery

### Objectives

- Prove the field-detection and PDF-rendering stack in the actual codebase.
- Define the canonical internal data model for documents, fields, and placements.
- Decide the default PDF stack:
  - viewer: `pypdfium2`
  - editor: `pikepdf`
  - OCR/searchability: `OCRmyPDF`
  - OCR/layout fallback: `Tesseract` or `docTR`

### Deliverables

- A small proof-of-concept that renders PDF pages and overlays candidate field boxes.
- A first-pass signature-field detector using heuristics and OCR hints.
- A placement data model that can store:
  - page number
  - bounding box
  - confidence
  - detected label
  - user confirmation state
- A short list of supported edit primitives:
  - rotate
  - merge/split
  - stamp signature
  - redact region
  - fill form field

### Acceptance Criteria

- A PDF page can be rendered consistently.
- At least one signature field candidate can be detected on sample forms.
- The current extraction flow remains unchanged.
- The proposal identifies any licensing or packaging issue before further work.

### Risks To Resolve Early

- PDF library licensing.
- Coordinate mapping between rendered view and PDF coordinates.
- OCR quality on scans.

## Month 2: MVP Build

### Objectives

- Integrate field detection into the desktop UI.
- Add reusable placement templates.
- Support saving a signed PDF with the new placement pipeline.
- Add searchable-PDF cleanup for scanned inputs.

### Deliverables

- "Auto-find signature fields" action in the PDF flow.
- Candidate overlay rendering with accept/reject behavior.
- Template save and reapply by document type or page size.
- Native PDF operations:
  - page rotate
  - merge/split
  - signature image stamp
  - form-field fill
  - redaction rectangle
- OCR cleanup path for image-only PDFs.

### Acceptance Criteria

- A user can open a PDF, find a likely signature location, place a saved signature, and export the result.
- Templates can be reused on another PDF.
- Searchable OCR output improves field detection on a scanned document.
- Saving a PDF does not corrupt the original input.

### Risks To Resolve

- False-positive field detection.
- Page-size drift across templates.
- Editing operations accidentally flattening data that should remain editable.

## Month 3: Hardening And Workflow Multipliers

### Objectives

- Add batch and watch-folder workflows.
- Improve ranking and confidence scoring for field detection.
- Start layout analysis and document quality scoring.
- Decide whether to extend into compliance-grade signing next.

### Deliverables

- Batch apply a template to multiple PDFs.
- Watch-folder job queue.
- Confidence scoring for detected fields.
- Quality checks for blur, skew, crop, and low contrast.
- Basic anomaly-style signature consistency checks.

### Acceptance Criteria

- The app can process more than one document without manual repetition.
- Users can inspect why a field was suggested.
- Low-quality scans are flagged before bad placements are exported.

### End-Of-Quarter Decision Point

At the end of 90 days, decide whether the next major investment is:

1. digital certificate signing,
2. cloud sync/team libraries,
3. browser extension/API,
4. or deeper layout intelligence.

## Recommended Execution Order

1. PDF rendering and coordinate mapping.
2. Field detection heuristics.
3. Placement templates.
4. Native edit/save pipeline.
5. OCR cleanup and layout analysis.
6. Batch and watch-folder automation.

## What Not To Do Yet

- Do not build a full generic PDF editor.
- Do not build cloud sync before the local workflow is strong.
- Do not add heavy ML models before the heuristic path is validated.
- Do not split the app into a second parallel signing system.
