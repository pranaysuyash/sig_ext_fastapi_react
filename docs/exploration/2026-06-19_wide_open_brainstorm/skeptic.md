# Skeptic

This is the Skeptic pass for the `wide open brainstorm` room.

Blunt version: most teams would overbuild this into a bloated PDF app, then wonder why users still only care about the one minute before export. The product wins only if it stays ruthlessly centered on repeated signature completion, not on clever surfaces, extra modes, or "AI" garnish.

## What users will ignore

- A big feature grid.
- Long explanations of confidence, model choice, or internal state unless they change the next action.
- Fancy map metaphors if the next click is still obvious.
- Generic AI summaries of what the user can already see.
- A sprawling signature library if the user only needs the last good signature and the last good placement.
- Anything that looks like admin work disguised as productivity.

Users do not buy a signature tool because it has more nouns. They use it because it gets them from source to export without friction, doubt, or rework.

## What adds noise without signal

- Over-detailed dashboards.
- Separate views for every tiny state.
- Too many cleanup sliders and quality knobs.
- Behavioral analytics that do not change placement or recovery.
- Chatty assistant layers that narrate the obvious.
- A settings graveyard for edge cases that happen once a quarter.

If a UI element does not either reduce rework, reduce fear, or reduce the number of decisions, it is probably noise.

## What should NOT be built

- A generic PDF editor.
- A cloud-first e-sign platform clone.
- A template marketplace.
- Team collaboration as a headline feature.
- A full document management system.
- A helpdesk disguised as AI.
- A "vault" that becomes a filing cabinet with a prettier icon.

Do not turn the product into a platform. This category is small enough that platform ambition is usually just self-inflicted drag.

## The common mistake products like this make

They optimize for the first capture and forget the second use.

That is the trap:

- the first signature import looks impressive,
- the first placement demo feels magical,
- the app then bloats into options, and
- the user still does not trust it enough to use it again next week.

Products like this also mistake "explanation" for "confidence." A user does not want a lecture about why a signature was chosen. They want the right thing to happen, fast, and to be able to undo it cleanly if it was wrong.

## What becomes local maxima instead of real differentiation

- Better OCR alone.
- More cleanup tools alone.
- Bigger signature libraries alone.
- A richer confidence score alone.
- More visual polish alone.
- More automation before the deterministic loop is boringly solid.

These feel like progress because they are measurable and demo-friendly. They are not differentiation if the user still has to do the same correction dance every time.

The local maximum is "we can detect signatures better." The real differentiation is "we can help the user finish recurring document work with less doubt on the second, third, and twentieth use."

## The 3 strongest concepts, if we stay restrained

### 1. Repeatable Completion Loop

The only loop that matters is: capture, place, review, export, reuse.

Why it survives skepticism:
- It is the whole job.
- It keeps the product from branching into unrelated tools.
- It measures value in closure, not feature count.

### 2. Proof Before Export

Nothing leaves the app without a visible reason to trust it.

Why it survives skepticism:
- Export is where mistakes become expensive.
- Users do not need more certainty theater; they need a simple, inspectable check.
- Recovery matters more than perfect automation.

### 3. Quiet Memory

Remember only the minimum needed to reduce rework on the next document.

Why it survives skepticism:
- A huge memory layer becomes maintenance overhead.
- Most users want repeated relief, not a digital archive project.
- The product should remember enough to help, not enough to become a system admin task.

## 6 / 12 / 24 month horizon

### 6 months

Make the core loop boringly fast.

- Faster source-to-placement flow.
- Fewer manual adjustments.
- Better defaults.
- Cleaner recovery when placement is wrong.
- No new product surface unless it directly shortens the loop.

### 12 months

Add memory only where it proves reuse.

- Recognize recurring document families.
- Reuse the last good setup more often.
- Make corrections feed future defaults.
- Keep the interface small enough that people do not need a guide.

### 24 months

The mature version should assume repeat use, not experimentation.

- The app knows the user's common signing patterns.
- Recovery is first-class.
- Auditability exists because it helps trust, not because it sounds enterprise.
- The product is a reliable completion instrument, not a document theme park.

## One non-obvious insight

The first capture is not the product's real test. The first reopened file is.

If the user can come back later, see the same state, and finish in a way that feels safe, the product is useful. If not, everything else is decoration.

That is the part most teams miss in a wide open brainstorm: the product is not winning on creation. It is winning on return.

## Final restraint

The right question is not "what else can we add?"

It is:

- what do users actually notice,
- what will they never forgive,
- and what can we leave out without weakening the core loop?

If the answer is "a lot," good. That means the product still has a chance to be sharp.
